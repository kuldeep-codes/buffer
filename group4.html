You are using EDTs whenever possible in coding for Finance and Operations. Which of these statements is not a benefit of using EDTs?

  • A. EDTs are more easily compiled
  • B. EDTs are reusable throughout the system

    A. EDTs are more easily compiled

While EDTs are a fundamental best practice in F&O development, the benefit does not lie in the compilation process itself. In fact, the compiler has to do more work to resolve the EDT's properties and its inheritance hierarchy compared to a simple primitive type like str or int.

The real benefits of using EDTs are:

  • B. EDTs are reusable throughout the system: This is a primary benefit. You define an EDT once (e.g., CustAccount) and use it for table fields, class variables, and method parameters everywhere a customer account is needed. This ensures consistency.
  • C. The code is easy to read: Using EDTs makes code self-documenting. void myMethod(CustAccount _custAccount) is much clearer and less error-prone than void myMethod(str _accountNum).
  • D. Properties can be inherited from parent EDTs: This is a powerful feature for maintainability. If you create a CustName EDT that extends a base Name EDT, CustName will inherit properties like StringSize. If you need to change the length of all name fields, you can often do it in one place (the base EDT).
