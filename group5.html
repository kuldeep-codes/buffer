  • Action
  • Display
  • Output

Answer Area

Process

  • Process1
  • Process2
  • Process3
    | Process | Menu item type | Justification |
    | :--- | :--- | :--- |
    | Process1 | Action | A recurring, automated process is a batch job, which is invoked by an Action menu item linked to a runnable class. |
    | Process2 | Display | Manually reviewing and correcting data requires a user interface (a form). Display menu items open forms. |
    | Process3 | Action | An export to another system is best modeled as a "process" or "action", as it often involves more than just generating a file. |

Explanation

  • Process1: Automatically check new transactions against validation rules every hour.

    • Correct Type: Action
    • This describes a batch job or a recurring process that executes business logic without direct user interaction. Action menu items are used to invoke runnable classes that perform these kinds of processing tasks. A user would use this menu item to set up and schedule the recurring job.

  • Process2: Manually review and correct transactions that have errors. No preprocessing is done.

    • Correct Type: Display
    • This requires a user to interact with data on a screen. Display menu items are used to open forms, which provide the user interface for viewing, editing, and correcting records.

  • Process3: Export transactions to another system when they successfully post.

    • Correct Type: Action

An "export to another system" is rarely just about creating a file. It is a full integration process that often includes:

  • Complex Logic: Transforming data in ways that are difficult or inefficient in a report's data provider.
  • Communication: Actively sending the data, for example, by calling a REST API or connecting to an SFTP server. An Output menu item cannot do this; it only generates a file for the user to download or saves it to an archive.
  • State Management: After a transaction is successfully exported, you typically need to update its status in the database (e.g., mark it as "Exported"). This is a data-writing operation, which is the job of a process, not a report.

you often "can't perform it by output report" because a real-world export is more than just a report. It's a runnable process. Therefore, you would create a class to handle all this logic, and you would use an Action menu item to trigger that class.

Explanation

  • Primary Index: Its main purpose is to enforce entity integrity by ensuring every record in the table has a unique identifier. By definition, a primary index does not allow duplicate values and is used as the default lookup key for a record.
  • Clustered Index: This index determines the physical storage order of the data on disk. Since the table's rows can only be physically stored in one order, a table can only have one clustered index. By default in Dynamics 365, the primary index is also the clustered index unless specified otherwise.
  • Non-Clustered Index: This index is a separate structure from the data rows that contains pointers back to the actual data. It is used to improve the performance of queries by providing a fast lookup path for data based on the indexed columns. You can have multiple non-clustered indexes on a single table.

      | Requirement | Option |
      | :--- | :--- |

Explanation

  • Track and delete only insert statements -> LegacyRollback

    • The LegacyRollback mode is a specific implementation from older versions of the SysTest framework. Its mechanism is to keep a list of all records that are created (inserted) during the test's execution. At the end of the test, it iterates through this list and issues a delete command for each record. It explicitly does not track any update or delete operations, so those changes will be committed to the database. This perfectly matches the requirement to only track and delete inserts.

  • Roll back transactions by using SQL save points -> AutoRollback

    • AutoRollback is the modern and recommended transaction mode. It works by creating a SQL transaction savepoint before the test method runs. After the test completes, it simply issues a command to roll back to that savepoint. You need to use the Extensible Data Security (XDS) framework to restrict to the data in the Sales table form based on the customer group and the role of the user.

Which three actions should you perform in sequence? To answer, move the appropriate actions from the list of actions to the answer area and arrange them in the correct order.

Actions:

  1. Create a security policy that links to the query and primary table
  2. Create a query to define filters
  3. Define the context for the rule.
  4. Define the list of constrained tables
  5. Assign the newly created role to the user
  6. Create new user role and add required privileges.
  • Create a query to define filters
    → This query defines the filtering condition (e.g., filter SalesTable based on CustomerGroup).
  • Create a security policy that links to the query and primary table
    → This links the query to the table you want to secure (SalesTable).
  • Define the context for the rule
    → Define the policy context, usually based on role or user scope. This is how the rule activates only for certain roles/users.

❌ Actions Not Required in XDS Setup Flow:

  • Create new user role and add required privileges
    → May be necessary in general, but not specific to the XDS configuration itself.
  • Assign the newly created role to the user
    → Happens outside the XDS policy configuration — done by admins when assigning roles, not part of the XDS logic building.
