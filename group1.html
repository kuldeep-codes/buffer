Correct Answers

C. Reference the Application Foundation package when creating the extension model

  • Why it's correct: This is a mandatory, non-negotiable step. To extend an object (like a table, form, EDT, or class) from any package, your model must explicitly reference that package. The compiler uses this reference to find the original object's definition. Without the reference to the Application Foundation package, you would not be able to see or extend any of its objects. This is the most critical step for enabling the customization itself.

D. Create a new model that builds into its own separate assembly

  • Why it's correct: This is a fundamental best practice for modern development in Dynamics 365. When you create a new model, the wizard asks if you want to "Create new package" or add the model to an existing package. Choosing "Create new package" ensures your model compiles into its own DLL (assembly). This provides:
    • Isolation: Your customization is a self-contained unit.
    • Modularity: It can be deployed, serviced, or removed independently.
    • Maintainability: It clearly separates your custom code from standard Microsoft code and other ISV solutions.
      This is a key configuration step that defines the architecture of your solution.

Analysis of the Other Options

B. Assign the model to the USR layer

  • Why it's less correct than D: While assigning your customization model to the USR (User) layer is the correct best practice for code ownership, it is arguably less fundamental than the architectural decision in option D. The layer system is crucial for managing code and preventing conflicts, especially in the (now rare) case of overlayering. However, for a pure extension model, the decision to create a new, separate package/assembly (D) has a greater impact on the solution's architecture, build process, and deployment. The question asks for two actions to "create and configure a new model," and both C and D are core decisions made in the model creation wizard that define what it can do and how it is structured.

A. Create an extension class that references the Application Foundation

  • Why it's incorrect: This is an action you perform after the model has been created and configured. It is a development task that takes place within the correctly configured model. It is not part of the initial model setup itself.

E. Create a new model that is part of an existing package

  • Why it's incorrect: This is an outdated practice that is strongly discouraged. Adding your custom model to a standard package (like Application Suite) creates a tight coupling, making deployments and updates much more complex and risky. It violates the principle of keeping customizations isolated.

You need to explain the performance advantages of the different concurrency models.

What are three performance advantages of optimistic concurrency control over pessimistic concurrency control? Each correct answer presents a complete solution.

Options:
A. One record cannot be selected for update by two different processes at the same time.
B. Records are locked as soon as they are retrieved from the database.
C. Records are locked for a shorter length of time.
D. Fewer resources are used to lock records during updates.
E. Records remain available for other processes while they are selected from the database.
‚úÖ C. Records are locked for a shorter length of time.

  • With optimistic concurrency, a record is only locked at the very moment the system is attempting to save an update or deletion. This is a brief, transactional lock. In contrast, pessimistic concurrency locks the record as soon as it is fetched for an update and holds that lock until the transaction is completed or rolled back. This extended lock duration in the pessimistic model can lead to significant delays for other users or processes needing to access the same record.

‚úÖ D. Fewer resources are used to lock records during updates.

  • Since pessimistic concurrency holds locks for a longer duration, it consumes more system resources to maintain these locks. This can include memory and database overhead. The optimistic model, by only locking for a very short period, is more resource-efficient, allowing the system to handle a higher volume of concurrent transactions with less overhead.

‚úÖ E. Records remain available for other processes while they are selected from the database.

  • A major performance bottleneck of pessimistic concurrency is that once a record is selected for an update by one process, it becomes unavailable to other processes. This can cause cascading wait times and reduce overall system throughput. Optimistic concurrency allows multiple processes to read and even intend to update the same record simultaneously. The system only checks for conflicting updates at the moment of saving. This high degree of availability keeps data flowing and processes moving, which is particularly beneficial in high-volume environments.

Incorrect options:

‚ùå A. One record cannot be selected for update by two different processes at the same time.

  • This describes pessimistic concurrency, where locking prevents other processes from updating the record.

‚ùå B. Records are locked as soon as they are retrieved from the database.

  • Again, this is pessimistic behavior. Optimistic concurrency delays locking until changes are committed.
